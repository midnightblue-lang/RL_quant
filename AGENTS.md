# 项目协作规则

本文件给后续 Codex agent 使用。开始任何实现前，先阅读本文件和 `PROJECT_PLAN.md`。如果两者冲突，优先遵守本文件中的工程纪律，再回到 `PROJECT_PLAN.md` 的阶段计划中执行。

## 当前阶段重点

- 当前阶段以**数据链路、回测框架、交易约束和绩效评估**为主。
- 不要一步登天接入真实交易、模拟盘、券商 API、自动下单或账户管理。
- 不要把 FinRL、Qlib、RD-Agent 等大型框架直接引入为项目主依赖；它们只能作为设计参考，除非用户明确批准。
- RL 公式因子生成和 XGBoost 截面预测是后续研究阶段，不应抢在基础回测和数据口径稳定之前实现。

## 代码风格

- 保持代码简洁、直接、可读。优先写清楚的小函数，不要提前设计庞大抽象。
- 不要为了“看起来稳健”添加大量无用兜底机制。只处理当前阶段真实会遇到的错误。
- 避免宽泛的 `except Exception`。如果必须捕获，必须说明原因，并保留足够的错误上下文。
- 不要静默吞错。数据、回测、标签、交易约束相关错误宁可 fail fast，也不要悄悄跳过。
- 不要写“万能配置层”“万能适配器”“插件系统”这类当前用不到的结构。
- 不要复制粘贴大段相似逻辑；如果重复出现 3 次以上，再考虑提取函数。
- 中文文档为主，代码变量、函数、类名使用清晰英文。
- 注释只解释不明显的业务口径、无未来函数约束或金融计算假设，不写机械注释。

## 测试与临时代码

- 临时 smoke test、一次性 debug 脚本、探索性 notebook 代码在验证完成后必须删除，或整理成正式测试/正式命令。
- 不要长期保留 `scratch.py`、`tmp_check.py`、`test_smoke_manual.py` 这类临时代码。
- 如果需要 smoke pipeline，只保留一个明确、文档化、可重复执行的入口；其他临时代码要清理。
- 核心逻辑必须优先写可复现单元测试，尤其是：
  - 交易日和调仓日计算。
  - 未来收益标签。
  - 无未来函数边界。
  - 涨跌停、停牌、买卖失败处理。
  - 成本扣减。
  - 权重归一化。
  - 净值和回撤计算。
- 测试应小而确定，不依赖大规模网络下载或长时间训练。

## 回测规则

- 当前项目只做研究回测，不做实盘交易。
- 月末信号只能在下一交易日开盘成交，不能用当日收盘价假装已成交。
- 不允许未来函数。任何 rolling、标签、股票池筛选、训练窗口都必须明确只使用当时可见数据。
- 回测中必须显式处理：
  - 停牌不可交易。
  - 涨停不可买。
  - 跌停不可卖。
  - 开盘价缺失不可成交。
  - 买入失败资金保留为现金。
  - 卖出失败持仓继续保留。
  - 买入和卖出成本分开配置。
- 绩效指标至少包括累计收益、年化收益、年化波动、最大回撤、Sharpe、Calmar、换手率、交易成本。
- 不要为了让结果好看而弱化交易约束或忽略成本。

## 因子挖掘规则

- 可以主动查找更多量化因子、公式 alpha、symbolic regression、program synthesis、强化学习公式生成相关论文，寻找更合适的算子、表达式约束、奖励函数和因子筛选方法。新增算子前必须记录来源、直觉、输入输出类型和引入原因。
- 必须注意因子的量纲问题，避免生成或保留量纲上无意义的表达式。例如价格可以和价格相减，价格除以价格可以得到无量纲比例，但不要随意把价格、成交量、成交额、时间窗口、常数混合成无法解释的量。表达式系统应尽量维护特征和算子的单位/类型约束。

## 数据与依赖

- 大规模行情数据、模型文件、缓存和报告输出不要提交进 git。
- 数据下载必须支持缓存和断点续传，但不要写过度复杂的多数据源 fallback。
- AKShare 是第一版默认数据源。除非用户明确要求，不要同时接入 Tushare、BaoStock、Qlib 数据格式。
- 网络依赖和安装依赖前要说明原因。不要悄悄安装大型包。
- 所有实验配置应进入 `config/*.yml`，不要把关键参数散落在 notebook 或脚本中。

## Agent 协作方式

- 如果 multi-agent/subagent 工具可用，复杂任务应拆给子 agent 并行处理。
- 适合拆分的子任务：
  - 数据质量检查。
  - 回测交易约束审查。
  - 绩效指标手算校验。
  - 代码简化和重复逻辑检查。
  - 无未来函数审查。
  - 文档与 README 校对。
- 子 agent 的产出必须由主 agent 复核后再合并。不要直接照抄未经验证的建议。
- 子 agent 不应执行真实交易、安装大依赖或修改项目边界。
- 如果任务很小，不要为了形式感创建子 agent。

## 实现顺序

优先顺序如下：

1. 项目骨架和配置。
2. 数据 schema、缓存和质量检查。
3. 股票池和标签。
4. 回测引擎、交易约束、成本、绩效指标。
5. 因子表达式系统。
6. alpha pool。
7. RL 公式生成。
8. XGBoost 截面预测。
9. 报告和复盘。

当前短期不要跳到第 7 或第 8 步。

## 外部 skill / 资源调研记录

以下资源可作为后续参考，但不要未经用户确认直接引入为依赖：

- OpenAI 官方 `openai/skills` 仓库已标注 deprecated，并提示当前 Codex skill/plugin 示例应参考 `openai/plugins`。
  - https://github.com/openai/skills
  - https://github.com/openai/plugins
- `openai/plugins` 是官方 Codex plugin 示例仓库，可用于学习插件、skill、agent 规则组织方式。
- Microsoft Qlib 是 AI-oriented 量化投资平台，覆盖数据处理、模型训练、回测等完整链路，适合作为架构参考。
  - https://github.com/microsoft/qlib
- Microsoft RD-Agent 包含量化因子和模型优化的 multi-agent 研究方向，适合作为“不要第一版就过度自动化”的反面提醒和长期参考。
  - https://github.com/microsoft/RD-Agent
- FinRL 是金融强化学习研究框架，适合后续理解 RL 交易环境，但当前项目不应接入其 live trading 或生产交易栈。
  - https://github.com/AI4Finance-Foundation/FinRL
- vectorbt 和 backtrader 可作为回测 API 设计参考，但第一版优先自建一个最小、透明、可解释的月度截面回测器。
  - https://github.com/polakowo/vectorbt
  - https://github.com/mementum/backtrader

## 禁止事项

- 不要接入真实交易。
- 不要保存或要求用户提供券商账号、交易密码、API key。
- 不要把研究回测包装成投资建议。
- 不要为了性能结果牺牲可解释性和无未来函数约束。
- 不要保留完成后无用的临时 smoke/debug 文件。
- 不要把大数据文件、模型 checkpoint 或下载缓存提交到仓库。
